# DeepFrost — 冷库多智能体调度系统

基于 **LangChain + DeepSeek V4** 的多智能体系统，管理 **5 个独立冷库** 与 **8 台共享压缩机**，通过智能调度实现温度合规、能耗优化与设备均衡运行。

---

## 背景介绍

某冷库园区拥有 5 个独立冷库，共享一组 8 台压缩机的制冷机组。各冷库的冷量需求随时间显著变化：
- **白天**：进出货频繁，开门次数多，热扰动大，温度上升快；
- **夜间**：环境温度降低，进出货停止，需求稳定在较低水平。

传统固定规则调度难以实时应对多库争抢制冷资源、压缩机能效差异和运行时长不均等问题。本系统通过三个类型的自治智能体协同决策，在满足所有冷库温度硬约束的前提下，**最小化总能耗**并**均衡各压缩机的累计运行时长**。

---

## 解决的问题

1. **制冷资源冲突**  
   多个冷库同时需要大量制冷且压缩机总能力有限时，如何动态分配制冷量，保证最紧急的冷库优先获得资源。

2. **能耗与设备均衡**  
   压缩机在高能效区间运行、避免频繁启停，同时使各压缩机累计运行时长尽量均衡，延缓设备老化。

3. **多目标实时决策**  
   在冷库温度、压缩机状态、环境温度、电价等多因素动态变化下，实时生成兼顾安全、经济、节能的调度方案。

## 环境要求与启动

### 依赖
- Python 3.9+

### 安装
```bash
pip install -r requirements.txt
```

### 配置 DeepSeek API Key
在 `config.py` 中设置，或通过环境变量 `DEEPSEEK_API_KEY` 注入。

### 运行
```bash
# 白天高负荷，模拟 2 个周期
python main.py --scenario day --cycles 2

# 夜间稳定，模拟 2 个周期
python main.py --scenario night --cycles 2
```

### 命令行参数
- `--scenario`：选择 `day` 或 `night`，默认 `day`
- `--cycles`：模拟周期数，每周期代表 15 分钟

## 核心技术说明

### 多智能体架构

系统共 **7 个自治智能体**：

| 智能体类型 | 数量 | 代表实体 | 核心职责 |
|------------|------|----------|----------|
| **CSA (冷库智能体)** | 5 | 每个独立冷库 | 感知温度，预测热扰动，计算安全/经济冷量，提交带优先级的标书 |
| **CCA (压缩机集群智能体)** | 1 | 8台压缩机集群 | 汇总所有冷库需求，结合压缩机状态、能效和累计运行时求解最优调度并下发 |
| **CDA (协调与数据服务智能体)** | 1 | 环境与市场 | 提供室外温度、电价；供需紧张时主持拍卖、生成影子价格 |

评估功能由独立工具模块（`evaluation.py`）提供，其结果通过记忆注入 CCA 形成反馈闭环。

### ReAct执行模型

每个智能体均采用 **OpenAI Function Calling** 模式（兼容 DeepSeek），LLM 根据角色提示、历史记忆和用户指令，自主决定调用哪些工具及参数。

### 工具层设计

所有数值计算与业务逻辑封装为独立的工具函数，LLM 仅负责调度，不参与具体计算。

| 工具名称 | 归属 Agent | 功能 |
|----------|-----------|------|
| `read_room_sensor` | CSA | 读取指定冷库的当前温度与允许区间 |
| `predict_disturbance_load` | CSA | 根据时间、进出货次数、活动程度预测热扰动 (kW) |
| `calculate_cooling_demand` | CSA | 基于热力学模型计算安全冷量 (Q_safe) 和经济冷量 (Q_eco) |
| `submit_cooling_demand` | CSA | 向全局 `demands` 字典提交制冷需求标书 |
| `read_compressor_status` | CCA | 读取所有压缩机的启停、负荷率、累计运行小时数 |
| `solve_optimal_schedule` | CCA | 贪心求解最优调度方案（优先满足安全需求，按累计时长升序启用） |
| `set_compressor_schedule` | CCA | 向压缩机模型下发启停及负荷指令 |
| `read_environment` | CDA | 获取当前室外温度与电价 |
| `update_environment` | CDA | 更新环境参数 |
| `resolve_auction` | CDA | 供需失衡时进行优先级拍卖，返回调整建议 |

每个工具调用均记录完整日志，包含参数与返回值，便于调试与演示。

### 物理模型

- **冷库热力学模型** (`thermal.py`)：一阶模型，热容量 `C = 20000 kJ/°C`，环境得热 5 kW，温度更新公式 `ΔT = (Q_in - Q_out) * dt / C`
- **压缩机模型** (`compressor.py`)：包含随负荷率变化的 COP 曲线，最大功率 100 kW，可计算当前电功率和制冷输出

## 测试数据说明

系统从 `tests/` 目录下的 JSON 文件自动加载场景，无需硬编码。提供两种典型工况：

### 白天高负荷 (`tests/test_day.json`)
- 室外温度 32.5°C，电价 0.85 元/kWh
- 5 个冷库初始温度在 -16.8°C ~ -17.8°C（接近上限 -15°C），活动程度 `high`
- 8 台压缩机累计运行小时 320h ~ 360h，存在明显不均衡
- 预期：冷量需求大，资源争抢激烈，需同时优化能耗与均衡度

### 夜间稳定 (`tests/test_night.json`)
- 室外温度 18.0°C，电价 0.35 元/kWh
- 初始温度 -18.5°C ~ -19.2°C（接近下限 -20°C），活动程度 `low`
- 压缩机累计运行小时同上，已存在不均衡
- 预期：需求较小，系统倾向让运行时间短的压缩机优先工作或整体停机节能

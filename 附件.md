## 附件：AI增强、协作、排错与模块说明

---

### 一、AI 增强点

本系统在三个关键环节利用 DeepSeek V4 实现了传统调度方法难以做到的智能决策：

**1. 冷库需求预测与动态优先级生成**

传统做法依赖固定阈值或 PID 控制，无法综合多因素预判需求。CSA 将当前温度、时段特征、进出货次数、活动程度等上下文以自然语言描述，由 LLM 推理生成：
- 安全冷量（Q_safe）：防止超温的物理下限
- 经济冷量（Q_eco）：趋近最佳储存温度的期望值
- 动态优先级（0~1）：综合温度紧迫度和业务因素

示例：温度 -16.8°C 接近上限 -15°C，白天高活动 +2次进出货，LLM 推理后生成优先级 0.64、安全冷量 60 kW、经济冷量 77.5 kW，而非固定值。

**2. 多目标优化调度的启发式引导**

CCA 在求解调度方案时，LLM 先生成一个兼顾能耗与运行均衡的初始可行解，再由确定性工具（贪心算法）微调。LLM 的常识推理能考虑“这台压缩机刚停，最好别马上再开”等启停约束，大幅缩减搜索空间。

**3. 供需失衡时的拍卖与影子价格**

当总需求超出压缩机最大能力时，CDA 通过 LLM 推理各冷库的影子价格（边际价值），引导 CSA 修正经济冷量需求，实现全局价值最大化。影子价格不是固定电价，而是由供需紧迫度决定的虚拟价格。

### 二、AI 协作说明

本项目的完整开发过程中，DeepSeek V4 作为协作工具参与了以下环节：

| 阶段 | 协作内容 |
|------|----------|
| **需求分析** | 向 DeepSeek 询问冷库热力学参数、压缩机 COP 曲线典型值，快速建立物理模型 |
| **架构设计** | 讨论多智能体分工方案，确定 CSA/CCA/CDA 的职责边界和通信方式（共享数据结构） |
| **代码生成** | 关键函数框架由 DeepSeek 辅助生成，本人进行工程化整合、错误处理、参数调优 |
| **测试场景设计** | 生成边界场景（如全部冷库同时要求满负荷、夜间零需求等）验证系统鲁棒性 |
| **排错辅助** | 分析错误日志（如 JSON 解析失败、温度异常等），提供修复思路和代码片段 |
| **文档撰写** | README 和本附件的初稿由 DeepSeek 辅助生成，本人调整技术细节与准确性 |

所有 LLM 辅助生成的内容均经过人工审查和测试验证，确保正确性和可靠性。

### 三、真实排错记录

**排错 1：LangChain ReAct 文本解析持续失败**

- **现象**：频繁出现 `Invalid Format: Missing 'Action:' after 'Thought:'`，工具从未被真正调用，压缩机始终停机，温度飙升。
- **排查**：通过日志发现 DeepSeek 的文本输出在“最终回答”后仍继续生成“思考-行动”序列，与 LangChain 原生 ReAct 解析器的预期格式不完全兼容。
- **解决**：将智能体基类从 `create_react_agent`（文本解析）全面改为 `ChatOpenAI.bind_tools()`（OpenAI Function Calling 模式）。LLM 通过 API 的 `tool_calls` 字段直接返回结构化参数，彻底消除文本解析错误。工具调用成功率从 0% 提升至 100%。

**排错 2：冷库温度异常过冷（-90°C 以下）**

- **现象**：白天场景下冷库温度骤降至 -93°C 以下，远超设定范围 -15°C ~ -20°C。
- **排查**：检查物理仿真发现两处问题——(1) `ColdRoomThermal` 热容量初始值 `C=500` 过小，制冷功率微小变化引起剧烈温度波动；(2) `calculate_cooling_demand` 工具中的经济冷量计算公式不合理，用“温度偏差 × 25”叠加扰动值，产生了远大于需求的制冷请求。
- **解决**：(1) 将热容量调整为 `C=20000`，符合真实冷库物理尺度；(2) 重写 `calculate_cooling_demand`，使其基于热力学公式 `Q_eco = Q_base + (C × ΔT) / dt` 计算真实所需冷量。修复后温度稳定在 -17°C ~ -18°C，完全合规。

**排错 3：部分周期冷库需求未被真正提交**

- **现象**：周期 3 以后 CCA 报告“没有收到任何冷库需求”，但 CSA 的日志输出声称已提交。实际制冷量为 0，压缩机空转。
- **排查**：发现 LLM 在最终回答中自行总结了提交结果，但 `submit_cooling_demand` 工具并未被真实调用，`demands` 字典为空。
- **解决**：(1) 将 `BaseToolCallingAgent` 的 `max_iterations` 从 8 增大至 12，给 LLM 足够的步数完成全部工具调用；(2) 在 CSA 的任务提示中增加“每一步都必须真实调用工具，不要跳过任何工具”；(3) 在周期末尾增加检查，若 `demands` 数量不足则记录警告。

### 四、本人负责的模块

| 模块 | 文件 | 具体内容 |
|------|------|----------|
| **整体架构设计** | 全部 | 确定 CSA/CCA/CDA 三类智能体分工，共享数据结构 + 记忆注入的协作模式 |
| **智能体基类** | `agents/base_agent.py` | 基于 LangChain + DeepSeek V4 的 Function Calling 基类实现（Plan-Act-Memory） |
| **工具层** | `tools.py` | 全部 10 个工具函数的设计、实现与日志记录，包含热力学计算、调度优化、拍卖协调 |
| **物理模型** | `models/thermal.py`, `models/compressor.py` | 冷库一阶热力学模型，压缩机 COP 曲线与能效模型 |
| **记忆模块** | `memory.py` | 滑动窗口记忆实现，支持 `save_context` 兼容接口 |
| **评估模块** | `evaluation.py` | 独立评估工具集，计算温度合规、能耗、均衡度等指标，生成自然语言反馈 |
| **命令行入口** | `main.py` | 场景加载（从 JSON）、主循环编排、物理仿真、评估反馈闭环 |
| **日志系统** | `utils/logger.py` | 统一日志（控制台 + 文件），所有工具调用和状态变更可追溯 |
| **测试场景** | `tests/test_day.json`, `tests/test_night.json` | 白天高负荷与夜间稳定两种典型工况的测试数据 |
| **智能体提示词** | `agents/csa.py`, `agents/cca.py`, `agents/cda.py` | 各智能体的角色、约束和任务描述 |
